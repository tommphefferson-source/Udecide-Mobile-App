Two request families in services/

  • API-server-proxied (auth, representatives, quiz, news, polls, questionnaires, fact-check, pages) — go to https://udecide-api.onrender.com/api with the custom AUTHTOKEN header. Real upstream keys live on the server.
  • Direct-to-external (electionsApi, congressApi, legiscanApi) — call third-party APIs from the client with an EXPO_PUBLIC_* key and fall back to bundled mock data when the key is missing. Elections is live (key baked in since build 11); Congress/LegiScan still serve samples.

Auth model

The legacy backend uses a custom AUTHTOKEN header (NOT Authorization: Bearer). Login/signup return an auth_token; the app stores it and the api-server forwards it upstream. A 401 anywhere clears the session and returns the user to login.

Sign in with Google is fully implemented (server-verified OAuth) but hidden in the 1.0 release behind SHOW_GOOGLE_SIGNIN=false in login.tsx — the server OAuth client is not yet configured, and offering third-party login triggers Apple's guideline 4.8 requirement to also ship Sign in with Apple. Both return together in a future release.
Section 3

Production Hosting on Render

The api-server runs in production on Render at https://udecide-api.onrender.com, deployed automatically from GitHub. This replaced the original Replit-only setup, which had no public production URL (native apps received 403s from the Replit dev domain — the root cause of the early TestFlight login failures).

3.1 How deployment works

  • Blueprint: render.yaml at the repo root defines the service (type web, runtime node, rootDir . so the pnpm workspace installs correctly).
  • Auto-deploy: every push to main triggers a fresh build and deploy (autoDeploy: true). Server changes ship with a git push — no dashboard action needed.
  • Build: pnpm install --no-frozen-lockfile && pnpm --filter @workspace/api-server run build (esbuild bundles to dist/index.mjs).
  • Start: node --enable-source-maps artifacts/api-server/dist/index.mjs. Render injects PORT.
  • Health check: GET /api/healthz → {"status":"ok"}. Render gates deploys on it.

3.3 Build-environment gotchas (learned the hard way)

  • No corepack enable — Render builders have a read-only /usr/bin (EROFS). pnpm is provisioned via the root package.json "packageManager": "pnpm@11.5.2" field instead.
  • esbuild build script — pnpm treats unapproved build scripts as fatal in CI (ERR_PNPM_IGNORED_BUILDS). esbuild is declared in ignoredBuiltDependencies; its Linux binary ships via the kept @esbuild/linux-x64 optional dependency.
  • trust proxy — app.set("trust proxy", true) is required behind Render's load balancer so req.ip resolves to the real client IP (the AI rate limiter keys anonymous callers by it).
Free-tier cold starts: the current instance spins down when idle; the first request after sleep takes ~50 seconds. For store reviews and real traffic, upgrade the service to the Starter plan (always-on) or run an uptime pinger against /api/healthz.
Replit relationship: the Replit workspace remains usable for development (the .replit-artifact config still works there), but Render is the production system of record. The app's production builds point at Render via EXPO_PUBLIC_DOMAIN.

Section 6

Backend — api-server

Node + Express 5 + TypeScript (ESM, esbuild-bundled). Exists to (1) hide paid keys, (2) proxy the legacy CodeIgniter backend at https://52.45.60.139/WS and translate its envelope into clean Zod-validated REST, (3) avoid CORS. Deployed on Render (§3).

6.1 Routes (all under /api)

  • auth.ts — POST /auth/login, /auth/signup, /auth/profile, DELETE /auth/account, POST /auth/profile/photo (multipart, 8 MB cap) → legacy /user_login, /user_sign_up, /edit_profile, /delete_account.
  • googleOauth.ts — GET /auth/google/start & /callback, POST /auth/google/exchange & /register (server-verified; dormant while the feature is hidden).
  • representatives.ts — GET /representatives → Cicero; {live, officials}; 502 on upstream failure.
  • elections.ts — GET /elections, /elections/voter-info → Google Civic v2.
  • factCheck.ts — POST /fact-check → Gemini (server key, nonpartisan system prompt, per-user rate limit — §6.3). Returns {text, mock}.
  • polls.ts / news.ts / civics.ts / pages.ts / home.ts / health.ts — polls (live-or-mock), RSS news with SSRF guard, quiz/questionnaires/terminology/parties, legacy static pages, healthz.

6.2 Auth flows

Email/password: proxied to the legacy service; failures map to 401/400 with friendly messages. Google OAuth: full server-mediated flow (HMAC state, code exchange with the server-side secret, verified email required, one-time exchange codes) — implemented and tested, awaiting OAuth client configuration + Sign in with Apple before re-enabling in the app.

6.3 AI rate limiting (added July 2026)

  • Client: the Fact Checker screen caps at 10 questions per session (input disables with an explanatory banner; a new session starts when the screen is reopened).
  • Server backstop: /fact-check enforces 10 requests per 15-minute sliding window per user, keyed by SHA-256-hashed AUTHTOKEN (client IP fallback via trust proxy). Over-limit → HTTP 429 with a friendly message the chat UI surfaces. Verified live in production (requests 11+ receive 429 on both keying paths).
  • Tunable on Render via FACT_CHECK_LIMIT / FACT_CHECK_WINDOW_MINUTES — no deploy needed.

Build, Release & Store Status

7.1 The release pipeline (EAS)

EAS Build is the single store pipeline; production builds load the EAS environment (§5.3), so binaries always point at the Render backend. EAS project: @udecide-apps-organization/udecide. iOS submission is fully automated via an App Store Connect API key (ascAppId 6783990826, team DG74FTMU5Y) configured in artifacts/udecide/eas.json.

# from artifacts/udecide/

eas submit --platform ios --profile production --latest # → TestFlight / App Store Connect
# Android: download the .aab from the build page and upload in Play Console

Section 8

Known Issues & Gotchas — Read This First

  1. Render free-tier cold starts. ~50 s to wake after idle; first app request may time out. Upgrade to Starter (always-on) or ping /api/healthz. This is the most user-visible operational issue.
  2. Congress.gov & LegiScan keys not set. Federal and state bill tabs silently serve sample data in production. Add the two EXPO_PUBLIC_* keys in the EAS environment and rebuild for live data.
  3. Google sign-in is hidden, not removed. Flip SHOW_GOOGLE_SIGNIN in login.tsx only after (a) GOOGLE_CLIENT_ID/SECRET + PUBLIC_ORIGIN are set on Render, (b) the redirect URI https://udecide-api.onrender.com/api/auth/google/callback is registered in Google Cloud, and (c) Sign in with Apple is implemented (App Store guideline 4.8).
  4. Concurrent file churn. A background Replit agent also commits to this repo (it once re-added an unused RECORD_AUDIO permission). Always git fetch/rebase before pushing, and review git diff before each build.
  5. Legacy backend uses a self-signed certificate. https://52.45.60.139/WS is encrypted but not CA-authenticated (NODE_TLS_REJECT_UNAUTHORIZED=0 on the api-server). The app must never call it directly.
  6. Do not delete the reviewer demo account. Both stores' reviewers use the demo credentials registered in App Review Information / Play App access.
  7. Monorepo decoys. mobile/ and the root eas.json are not the product; install from the repo root; the minimumReleaseAge supply-chain guard stays on.
  8. AI cost control. The Fact Checker is capped client- and server-side (§6.3); if Gemini quota issues appear, lower FACT_CHECK_LIMIT on Render before touching code.

Getting started (new developer)

  1. pnpm install at the repo root.
  2. App: pnpm --filter @workspace/udecide run dev from artifacts/udecide (Expo dev server).
  3. Backend: pnpm --filter @workspace/api-server run dev (needs PORT; §3.2 secrets for live features) — or just use the production Render API.
  4. Typecheck everything: pnpm run typecheck at the root.
  5. Where things live: screens → artifacts/udecide/app/; data → artifacts/udecide/services/; theming → constants/colors.ts; backend routes → artifacts/api-server/src/routes/; deploy → render.yaml.
  6. Ship: build with EAS (§7.1), verify in TestFlight/Internal testing, then submit.
